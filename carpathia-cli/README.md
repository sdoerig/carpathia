# Description

Carpathia is a CLI tool that generates database access code from your schema. You write the Tera templates—it produces the code. Currently supports Postgres (MySQL & SQLite planned). Functional but in beta: use it, test it, and help shape its future! 🚀
# carpathia-cli — Rust Database Schema Code Generator

> Generate Rust database access code from PostgreSQL schemas using customizable Tera templates.

`carpathia-cli` is a command-line tool that connects to a PostgreSQL database, extracts its schema (tables, views, columns, constraints), and generates Rust code based on user-defined Tera templates. It intelligently skips re-generation if nothing has changed — thanks to a built-in hash-based cache.

Perfect for teams building type-safe, maintainable database layers in Rust without writing boilerplate by hand.

---

## ✅ Features

- ✅ Extracts **tables, views, materialized views**, and column metadata from PostgreSQL
- ✅ Generates code using **Tera templates** (`.tera` files)
- ✅ **Caching**: Only regenerates files when schema or templates change
- ✅ **Type mapping**: Map PostgreSQL types to custom Rust types via `carpathia_type_mapping.json`
- ✅ Supports **dry-run**, **debug output**, and **schema inspection**
- ✅ No external dependencies beyond Rust and PostgreSQL

## 🚀 Quick Start

### 1. Install

```bash
cargo install carpathia-cli
Or build from source:

bash
cd carpathia-cli
cargo build --release
2. Prepare a PostgreSQL Database
Ensure you have a PostgreSQL server running and a database with a schema (e.g., Pagila).

3. Create a Template Directory
Create a folder (e.g., ./templates/rust_lib) with .tera files:

bash
mkdir -p templates/rust_lib
Example template: templates/rust_lib/tables.user.rs.tera

rust
// Generated by carpathia-cli
#[derive(Debug, Clone)]
pub struct {{ table_name }} {
{{- for attr in table.attributes }}
    pub {{ attr.column_name }}: {{ attr.u_type }},
{{- end }}
}
4. Create a Type Mapping File
Create carpathia_type_mapping.json:

json
{
  "version": "0.1.0",
  "type_mapping": {
    "integer": { "u_type": "i32" },
    "text": { "u_type": "String" },
    "uuid": { "u_type": "Uuid", "u_import": "use uuid::Uuid;" }
  }
}
5. Run the Generator
bash
carpathia-cli \
  --db-host localhost \
  --db-port 5432 \
  --db-username postgres \
  --db-password postgres \
  --db-name carpathia \
  --output-directory ./generated \
  --template-directory ./templates/rust_lib \
  --carpathia-type-mapping-file carpathia_type_mapping.json
6. Inspect Generated Code
Generated files will appear in ./generated/ — e.g.:

markdown
./generated/user.rs
./generated/address.rs
./generated/mod.rs
🛠 Advanced Usage
Flag  Description
--print-schema  Print the extracted schema as JSON to stdout
--print-db-types  Print all detected PostgreSQL types and their defaults — useful to build your mapping file
--cache-modus BypassCache Force regeneration even if schema didn’t change
--execute-templates false Dry-run: parse schema but skip template rendering
--cache-file ./my-cache.json  Use a custom cache file path
💡 Tip: Use --print-db-types to auto-generate your initial carpathia_type_mapping.json.

📁 Template Naming Convention
Templates are categorized by prefix:

Prefix  Purpose Example
tables.*.tera Generate one file per table tables.user.rs.tera → user.rs
views.*.tera  Generate one file per view  views.user_summary.rs.tera → user_summary.rs
summary.*.tera  Generate one summary file with full schema  summary.mod.rs.tera → mod.rs
The suffix (e.g., .rs, .html) is preserved. Use {{ table_name }} and {{ table.attributes }} in templates.

🔒 Security Note
Avoid passing passwords via CLI arguments. Use environment variables instead:

bash
export DB_PASSWORD=your_password
carpathia-cli --db-password "$DB_PASSWORD" ...
Or use a .env file with dotenv (if supported in your build).
